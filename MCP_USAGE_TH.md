# คู่มือการใช้งาน Model Context Protocol (MCP) สำหรับ File Graph Visualizer

เอกสารนี้จะอธิบายวิธีการโต้ตอบกับเซิร์ฟเวอร์ MCP ของ File Graph Visualizer รวมถึงเครื่องมือที่มี วิธีการใช้งาน และสถานการณ์ตัวอย่างต่างๆ

## 1. การรันเซิร์ฟเวอร์ MCP

File Graph Visualizer สามารถเริ่มต้นในโหมดเซิร์ฟเวอร์ MCP ได้โดยใช้แฟล็ก `-mcp` ซึ่งจะทำให้มันรอรับคำสั่ง MCP ผ่าน Standard Input/Output (stdio)

```bash
go run main.go -mcp
```

หรือหากคุณมีไฟล์ executable ที่คอมไพล์แล้ว:

```bash
./file_graph_server.exe -mcp
```

## 2. การโต้ตอบกับเซิร์ฟเวอร์ MCP

คุณจะต้องมี MCP Client เพื่อโต้ตอบกับเซิร์ฟเวอร์ โดยส่วนใหญ่แล้วอินเทอร์เฟซของ LLM จะมีความสามารถของ MCP Client ในตัว หากต้องการโต้ตอบแบบโปรแกรม คุณสามารถใช้ไลบรารี MCP Client ในภาษาที่คุณต้องการ หรือใช้เครื่องมือ Command-line เช่น `mcp-cli` (หากมี)

เซิร์ฟเวอร์ MCP คาดหวังและส่งคืนอ็อบเจกต์ JSON ผ่าน stdio ตามข้อกำหนดของ MCP

## 3. เครื่องมือ MCP ที่มีให้ใช้งาน

เซิร์ฟเวอร์ MCP ของ File Graph Visualizer มีเครื่องมือดังต่อไปนี้:

### 3.1. `getNodeCount`

**คำอธิบาย**: ส่งคืนจำนวนรวมของโหนดไฟล์ที่พบในการสแกนครั้งล่าสุด
**Input**: ไม่มี
**Output**:
```json
{
  "count": 123
}
```
**สถานการณ์ตัวอย่าง**: LLM อาจใช้เครื่องมือนี้เพื่อดูภาพรวมขนาดของชุดข้อมูลอย่างรวดเร็วก่อนที่จะดำเนินการค้นหาข้อมูลที่ละเอียดขึ้น

### 3.2. `getHeartbeat`

**คำอธิบาย**: ตรวจสอบสถานะการทำงานของเซิร์ฟเวอร์
**Input**: ไม่มี
**Output**:
```json
{
  "status": "alive",
  "timestamp": "15:04:05"
}
```
**สถานการณ์ตัวอย่าง**: LLM หรือระบบ orchestrator สามารถใช้สิ่งนี้เพื่อตรวจสอบว่าเซิร์ฟเวอร์ File Graph Visualizer กำลังทำงานและตอบสนองหรือไม่

### 3.3. `getNodeStatus`

**คำอธิบาย**: ดึงข้อมูลรายละเอียดสำหรับโหนดไฟล์ที่ระบุตามพาธแบบสมบูรณ์
**Input**:
```json
{
  "path": "C:\dev\file_graph\main.go"
}
```
**Output (หากพบ)**:
```json
{
  "found": true,
  "node": {
    "id": "C:\dev\file_graph\main.go",
    "name": "main.go",
    "path": "C:\dev\file_graph\main.go",
    "is_folder": false,
    "size": 5120,
    "modified_at": "2026-03-11T10:30:00Z",
    "created_at": "2026-03-11T10:30:00Z",
    "size_last_3": 120,
    "name_last_4": "main",
    "hash": "a1b2c3d4e5f6...",
    "vector": [0.1, 0.2, 0.3, ...]
  }
}
```
**Output (หากไม่พบ)**:
```json
{
  "found": false,
  "node": null
}
```
**สถานการณ์ตัวอย่าง**: LLM หลังจากถูกถามเกี่ยวกับไฟล์เฉพาะ สามารถใช้เครื่องมือนี้เพื่อดึงข้อมูลเมตา (ขนาด, วันที่แก้ไข, แฮช) เพื่อให้คำตอบที่มีรายละเอียดแก่ผู้ใช้

### 3.4. `searchAndHighlight`

**คำอธิบาย**: ค้นหาโหนดไฟล์ที่มีสตริงคำค้นหาอยู่ในพาธหรือชื่อ หากพบโหนด จะมีการส่ง ID ของโหนดนั้นไปยังเว็บ frontend เพื่อไฮไลท์ในตัวแสดงภาพ
**Input**:
```json
{
  "query": "index.html"
}
```
**Output (หากพบ)**:
```json
{
  "found": true,
  "node_id": "C:\dev\file_graph\index.html",
  "message": "พบโหนดและตั้งค่าเพื่อไฮไลท์"
}
```
**Output (หากไม่พบ)**:
```json
{
  "found": false,
  "message": "ไม่พบโหนดที่ตรงกับคำค้นหา"
}
```
**สถานการณ์ตัวอย่าง**: LLM สามารถใช้เครื่องมือนี้เพื่อช่วยผู้ใช้ค้นหาไฟล์หรือไฟล์ที่เกี่ยวข้องในกราฟด้วยสายตาได้ ตัวอย่างเช่น หากผู้ใช้ถามว่า "แสดงไฟล์ HTML หลักให้ฉันดูหน่อย" LLM สามารถเรียกใช้ `searchAndHighlight` ด้วย "index.html" และโหนดนั้นจะถูกไฮไลท์ในเบราว์เซอร์

## 4. การสะท้อนใน UI

เมื่อเรียกใช้เครื่องมือ `searchAndHighlight` สำเร็จ อินเทอร์เฟซเว็บของ `File Graph Visualizer` (ที่ทำงานพร้อมกัน) จะตรวจสอบปลายทาง `/api/highlighted-node` เป็นระยะๆ หากมี `highlightNodeId` ส่งคืน UI จะเลือกโหนดนั้นโดยอัตโนมัติ แสดงรายละเอียด และเลื่อนมุมมองไปที่ศูนย์กลางของโหนดนั้น สิ่งนี้จะให้ข้อเสนอแนะภาพโดยตรงจากการโต้ตอบของ LLM ไปยังอินเทอร์เฟซผู้ใช้

## 6. AI ทำความเข้าใจเครื่องมือ MCP ได้อย่างไร ("คู่มือสำหรับ AI")

เพื่อให้ AI (โดยเฉพาะ Large Language Model หรือ LLM) เข้าใจและสามารถใช้เครื่องมือ MCP เหล่านี้ได้อย่างมีประสิทธิภาพ มันจะอาศัยข้อมูลหลักสองส่วนที่ได้รับโดยตรงจากเซิร์ฟเวอร์ MCP:

1.  **คำอธิบายภาษาธรรมชาติ (Natural Language Descriptions)**: เครื่องมือแต่ละชิ้นจะถูกลงทะเบียนพร้อมกับคำอธิบายที่ชัดเจนและเข้าใจง่ายสำหรับมนุษย์ (เช่น "ส่งคืนจำนวนรวมของโหนดไฟล์จากการสแกนครั้งล่าสุด") LLM จะใช้ความสามารถในการทำความเข้าใจภาษาธรรมชาติเพื่อจับคู่ความตั้งใจของผู้ใช้กับวัตถุประสงค์ของเครื่องมือนั้นๆ

2.  **Schema โครงสร้าง (Structural Schema - Input/Output Definitions)**: เซิร์ฟเวอร์ MCP ยังให้คำจำกัดความอย่างเป็นทางการ (schema ซึ่งมักอยู่ในรูปแบบ JSON Schema) สำหรับพารามิเตอร์อินพุตของเครื่องมือและเอาต์พุตที่คาดหวัง Schema นี้ซึ่งได้มาจากโครงสร้าง Go และแท็ก `json` / `jsonschema` ในโค้ดของเรา จะบอก LLM ว่า:
    *   เครื่องมือต้องการอาร์กิวเมนต์อะไรบ้าง (เช่น `path: string`)
    *   ประเภทข้อมูลของอาร์กิวเมนต์เหล่านั้นควรเป็นอะไร
    *   โครงสร้างของข้อมูลที่ส่งคืนจะเป็นอย่างไร

ดังนั้น "คู่มือ" ที่ AI อ่านไม่ใช่เอกสารแยกต่างหาก แต่เป็นข้อมูลเมตาที่มีโครงสร้างและสมบูรณ์นี้ที่เซิร์ฟเวอร์ MCP ให้มาแบบไดนามิกเกี่ยวกับความสามารถและอินเทอร์เฟซของเครื่องมือแต่ละชิ้น LLM จะประมวลผลข้อมูลนี้เพื่อตัดสินใจว่าจะเรียกใช้เครื่องมือใด วิธีสร้างอินพุต และวิธีตีความเอาต์พุต
